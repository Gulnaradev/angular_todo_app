# QA STRATEGY — Angular Todo App

## 1) Стратегия тестирования для команды 5 dev + 2 QA
### Роли и поток
- QA участвуют с начала: уточнение требований, acceptance criteria, риск-анализ, оценка задач
- Для каждой фичи: dev делает unit/component tests, QA добавляет/обновляет тестовую документацию и пишет E2E по критичным сценариям. 
- Перед релизом: короткий regression + exploratory

### Что внедрять в первую очередь (на примере задания)
1) Smoke/critical E2E (Playwright): smoke тесты на загрузку, Create/Update/Delete, фильтры.
2) Проверки на API-особенности (неперсистентность/задержки) — как edge tests.
3) Минимальный набор unit/component tests на бизнес-логику (trim/валидация/редьюсер/сервис).
4) Observability: логирование ошибок API, понятные UI-состояния (loading/error/empty).

## 2) CI/CD: интеграция автотестов
### На каждый коммит / PR
- Lint + typecheck
- Unit/component tests (быстро)
- Playwright: smoke + критичный CRUD (Chromium, headless)
Цель: быстрый feedback <10 минут.

### Перед релизом (release pipeline)
- Полный набор Playwright E2E
- Кроссбраузер (Chromium/Firefox/WebKit)
- Репорты (HTML/Allure при необходимости)
- Тестирование на “медленном API” (моки/инструментирование)

## 3) Метрики качества
- Defect leakage (сколько багов ушло в прод)
- Pass rate E2E / flaky rate (процент нестабильных тестов)
- Lead time to fix (время от заведения до закрытия)
- Coverage по критичным user flows (не “проценты строк”, а сценарии)
- MTTR для CI падений автотестов
- Количество инцидентов, связанных с интеграцией API

## 4) Развитие команды и онбординг QA
### Онбординг (1–2 недели)
- Доступы, среда, запуск приложения/тестов
- Обзор архитектуры, точек интеграции с API
- Гайд по тест-дизайну и баг-репортингу (шаблоны)
- Практика: добавить 1–2 теста + ревью с ментором

### Ключевые навыки QA 2024–2025
- Автоматизация E2E (Playwright/Cypress) + основы API testing
- Понимание CI/CD, умение читать логи, trace, видео
- Тест-дизайн, риск-ориентированный подход, работа с требованиями
- Основы web (DOM, сеть, http, async)
- Коммуникация: качество требований, приоритизация, severity/priority
