# TEST PLAN — Angular Todo App

## 1. Цель
Обеспечить качество и готовность к релизу Angular Todo App (создание/редактирование/удаление задач, отметка выполнения, фильтрация) при использовании внешнего API JSONPlaceholder с известными ограничениями.

## 2. Scope (что тестируем)
### In scope
- UI: отображение списка задач, создание, редактирование, удаление
- Изменение статуса выполнения (checkbox)
- Фильтры: All / Active / Completed
- Поведение при известных особенностях API:
  - изменения не сохраняются после refresh
  - POST возвращает объект с сгенерированным id
  - PUT/DELETE “успешны” для любых id
  - задержка ответа до 2 секунд

### Out of scope (для данного релиза/тестового)
- Нагрузочное тестирование в полном объёме (только базовые проверки производительности/задержек)
- Кросс-девайс UX/адаптив (можно дополнить при наличии требований)
- Pen-test/полная security-оценка
- Локализация (если нет требований)

## 3. Виды тестирования и обоснование
- Функциональное тестирование (manual + автотесты E2E): покрытие основных пользовательских сценариев.
- UI/UX sanity: проверка базовой понятности состояний (список/пустое/ошибка).
- Интеграционное тестирование взаимодействия с API: проверка корректности обработки ответов, задержек, ограничений API.
- Регрессионное тестирование: запуск набора E2E на ключевых сценариях при изменениях.
- Кроссбраузерное тестирование: Chromium/Firefox/WebKit (Playwright projects) — минимально для релиза.
- Exploratory testing: поиск дефектов и UX-проблем, особенно при ошибках/медленном API.

## 4. Критерии входа (Entry criteria)
- Приложение собирается и запускается локально (npm install, npm start)
- Доступен тестовый контур (локальный) + стабильный доступ к JSONPlaceholder
- Определены требования/ожидаемое поведение по edge cases API (например, “после refresh изменения пропадают” — ожидаемо)
- Согласованы браузеры для тестирования: Chromium/Firefox/WebKit

## 5. Критерии выхода (Exit criteria)
- Пройден минимальный набор критичных сценариев:
  - Create/Update/Delete
  - Toggle completed
  - Filters
- Нет дефектов severity Blocker/Critical открытых к релизу
- Автотесты E2E “зелёные” в целевых браузерах (минимум Chromium; желательно все 3)
- Оформлены тест-кейсы и баг-репорты по найденным дефектам
- Подготовлен краткий отчёт REPORT.md

## 6. Риски и митигации
### Риск: нестабильность внешнего API / сеть
- Митигация: использовать network mocking для части тестов (page.route) + ретраи/trace для диагностики.

### Риск: неперсистентность JSONPlaceholder приводит к “неожиданным” результатам
- Митигация: явные edge-case тесты + описание в документации/релиз-нотах.

### Риск: флаки из-за задержек до 2с
- Митигация: ожидания через expect(locator).toBeVisible с адекватными таймаутами; не использовать fixed sleep без необходимости.

### Риск: отсутствие стабильных локаторов
- Митигация: использовать семантические class-локаторы (task-input/task-item/filters) и рекомендация добавить data-testid в будущем.

## 7. Оценка трудозатрат (high-level)
Предполагаем команда: 2 QA + 5 dev.

- Анализ требований/рисков: 2–3ч
- Тест-дизайн (10–15 кейсов) + ревью: 2–3ч
- Exploratory testing + баг-репорты: 3-4ч
- Автоматизация E2E (базовый набор): 4–6ч
- Интеграция в CI + отчётность: 2–4ч
Итого: ~13–20ч (2–3 рабочих дня одним QA; быстрее при параллели).
